## 1. 目标与边界

### 1.1 P0 的唯一目标

将现有“PyQt 窗口直接在线程中调用采集/处理函数”的执行方式，改造成一套**可脱离 UI、可由命令行和未来 Worker 调用**的统一执行核心。

P0 完成后，以下三个入口必须能调用同一份业务执行逻辑：

```text
PyQt 工具窗口 ─┐
命令行 Host ───┼──> CollectorRunner ───> 现有采集器 / 数据处理逻辑
未来 Worker ───┘
```

### 1.2 P0 交付物

- `RunnerContext`、`RunnerRequest`、`RunnerEvent`、`CollectorRunner` 的稳定接口。
- 无 UI 的 `runner_host`，以 JSON 文件接收任务，以 JSONL 输出事件。
- Manifest 新增可选的 `runner_entrypoint`。
- 工具发现以 manifest 为唯一真源；静态 registry 不再保存人工维护的工具列表。
- 两个试点 Runner：
  - `xlsx_merge`：纯数据处理基线。
  - 一个 YouTube API 优先的轻量工具：优先选择 `profiles`（作者信息提取）；若其依赖拆分成本明显更低，也可由 `context` 替代。
- PyQt 原有窗口继续可用，并改为调用 Runner 或与 Runner 共享同一业务服务层。
- Runner、manifest、Host、试点工具和 PyQt 回归测试。

### 1.3 P0 明确不做

P0 **不**引入或实现以下内容：

- FastAPI、Web 管理端、PostgreSQL、Redis、NATS、Kafka。
- Worker Agent、任务领取、队列、调度、多机执行。
- 完整 schema 表单、参数模板数据库化。
- 浏览器池、会话共享、Context/Tab 调度。
- 统一结构化 item 入库、断点续采、任务重试策略。
- 一次性迁移全部 27 个工具。
- 大规模重写现有业务逻辑或改变已有 XLSX 字段。

P0 的判断标准不是“平台功能是否丰富”，而是“执行核心是否已经独立于 UI，并具备未来接入 Worker 的稳定边界”。

## 2. 当前代码基线与问题

### 2.1 当前结构

```text
main.py
└── src.studio.qt_app.main
    └── PyQt 主工具台
        ├── src.studio.discovery：扫描 *.manifest.json
        ├── src.studio.registry：静态 TOOLS 列表
        └── src.studio.tool_runner：QProcess 子进程启动工具窗口

src/ui/、src/platforms/*/windows.py、src/processing/windows.py
└── 采集参数表单、线程启动、日志展示、停止/暂停控制

src/platforms/*/*.py、src/processing/*.py
└── 实际采集或处理函数、XLSX 输出、浏览器/API 调用

src/core/
└── output、xlsx、browser、config_store、logging 等公共能力
```

### 2.2 当前可复用资产

- 工具 manifest 已覆盖 `src/platforms`、`src/processing`、`src/tools` 三个扫描目录。
- `ToolSpec` 已承担工具 ID、名称、分类、UI 入口、实现路径、标签等元数据。
- `tool_runner.py` 已有“独立进程运行单个工具”的思路，可作为未来 Host 的迁移参考。
- 多数任务已有 `log_callback`、`finish_callback`、`stop_event`、`pause_event` 等形态，适合通过适配器迁移。
- `src/core/output.py` 已提供统一输出根目录及平台子目录能力。
- 当前测试已覆盖工具发现、配置流、暂停状态机、部分 TikTok/X/YouTube 行为，可作为保护网。

### 2.3 P0 要解决的具体问题

1. **工具元数据双源**：`discovery.py` 扫描 manifest，但 `tool_runner.py` 仍优先使用手工维护的 `registry.TOOLS`。
2. **`ToolSpec` 没有 Runner 入口**：当前只描述 PyQt `entrypoint`。
3. **执行逻辑与窗口耦合**：UI 负责收参、启动线程、回调和产物提示，外部系统无法复用。
4. **输出不可统一观察**：日志、进度、文件产物、异常没有一致事件格式。
5. **进程契约缺失**：未来 Worker 无法稳定判断一个子进程的输入、事件、终态和退出码。

## 3. P0 完成后的目标形态

```text
┌──────────────────────────────┐
│  PyQt Window / CLI / Future  │
│        Worker Agent          │
└──────────────┬───────────────┘
               │ RunnerRequest
               ▼
┌──────────────────────────────┐
│        CollectorRunner        │
│ validate(request)             │
│ run(context) -> events        │
└──────────────┬───────────────┘
               │ RunnerEvent
               ▼
┌──────────────────────────────┐
│         Runner Host           │
│ JSON request -> JSONL events  │
│ terminal state + exit code    │
└──────────────┬───────────────┘
               │
               ├── stdout: 机器可读 JSONL
               ├── stderr: 诊断日志
               └── output/runs/<task_id>/: 产物
```

P0 的 Runner 只负责**执行一个任务**，不负责调度任务。未来 Worker Agent 只需要：

1. 从控制中枢获得任务 JSON；
2. 启动 `runner_host` 子进程；
3. 读取 stdout JSONL；
4. 将事件和产物元数据上报；
5. 根据退出码更新任务状态。

## 4. 关键设计决策

### 4.1 Manifest 是工具元数据唯一真源

- `*.manifest.json` 是工具名称、分类、UI 入口、Runner 入口和能力声明的唯一来源。
- 不再维护一份独立、硬编码的 `TOOLS` 工具清单。
- `registry.py` 在 P0 内保留为兼容层，但只能从 `discover_tools()` 构造结果，不能再人工枚举工具。
- 所有绑定均以 `tool_id` 为主键：配置、窗口、Runner、未来 Worker 能力、任务记录都不得以显示名称做关联。

### 4.2 UI 入口与 Runner 入口分离

- `entrypoint` 保持现有含义：PyQt UI 入口。
- 新增 `runner_entrypoint`：无 UI 的 `CollectorRunner` 类路径。
- 未迁移的工具允许没有 `runner_entrypoint`，仍可从 PyQt 启动；但它们在 P0 中必须被标记为 `runner_ready = false`。
- P0 试点工具必须提供可导入、可校验、可命令行执行的 Runner。

### 4.3 Runner 放入 `src/core/runner/`

Runner SDK 是全项目执行边界，不属于 PyQt，也不属于某个平台。因此其实现放在 `src/core/runner/`，避免未来 Web、Worker、CLI 反向依赖 `src/studio`。

### 4.4 JSONL 是进程边界协议

- `stdout` **只允许**输出一行一个 JSON 的 `RunnerEvent`。
- `stderr` 用于开发诊断、traceback 和人类可读日志。
- 不允许业务代码直接 `print()` 到 stdout。
- Host 统一负责事件序号、时间戳、终态事件和退出码。

### 4.5 Runner 不直接管理 PyQt 对象

Runner 层禁止导入：

```text
PyQt5
src.ui.*
src.studio.qt_app
QWidget / QMessageBox / QThread
```

Runner 可以使用业务层、`src.core` 公共模块、Playwright、Google API、openpyxl 等实际执行依赖。

### 4.6 P0 采用“薄 Runner + 业务服务层”

不要把所有旧函数强行塞进一个万能适配器。优先将试点工具分成：

```text
Window  -> 参数转换 -> Service/业务函数 -> 文件/API/浏览器
Runner  -> 参数转换 -> Service/业务函数 -> RunnerEvent
```

这样 PyQt 和 Runner 共享业务能力，但各自保留不同的输入输出适配层。

## 6. 核心接口契约

### 6.1 `RunnerRequest`：可序列化任务输入

`RunnerRequest` 是 Host、PyQt 和未来 Worker 之间传递的纯数据对象。它不得保存 Python 对象、线程对象、Qt 对象或回调函数。

```python
# src/core/runner/contracts.py
from __future__ import annotations

from dataclasses import dataclass, field
from pathlib import Path
from typing import Any, Literal, Mapping


@dataclass(frozen=True)
class EventPolicy:
    log_rate_limit_per_sec: int = 5
    metric_interval_ms: int = 1_000
    item_event_mode: Literal["off", "sampled", "full"] = "off"


@dataclass(frozen=True)
class RunnerRequest:
    protocol_version: int
    task_id: str
    tool_id: str
    params: Mapping[str, Any]
    config: Mapping[str, Any] = field(default_factory=dict)
    output_dir: str | None = None
    experiment_id: str | None = None
    run_id: str | None = None
    batch_id: str | None = None
    session_mode: str | None = None
    event_policy: EventPolicy = field(default_factory=EventPolicy)
```

约束：

- `task_id` 非空，格式建议为 `[a-zA-Z0-9_-]{1,128}`。
- `tool_id` 必须匹配 manifest 中唯一 ID。
- `params` 与 `config` 必须 JSON 可序列化。
- `output_dir` 缺省时，由 Host 自动生成 `output/runs/<task_id>/`。
- P0 不要求 `experiment_id`、`run_id`、`batch_id` 非空，但字段必须预留。
- P0 不在请求中传递浏览器 profile 的绝对路径；浏览器型工具继续从本地配置和既有逻辑获取。

### 6.2 `RunControl`：停止与暂停

```python
from threading import Event


@dataclass
class RunControl:
    stop_event: Event = field(default_factory=Event)
    pause_event: Event = field(default_factory=Event)

    def should_stop(self) -> bool:
        return self.stop_event.is_set()

    def wait_if_paused(self, interval_seconds: float = 0.2) -> None:
        while self.pause_event.is_set() and not self.stop_event.is_set():
            self.stop_event.wait(interval_seconds)
```

说明：

- `RunControl` 只存在于进程内，不从 JSON 传入。
- P0 Host 暂不实现远程暂停/停止 API；保留契约以便 PyQt 与 P1 Worker 复用。
- 命令行下可通过 `SIGINT` / `SIGTERM` 触发 `stop_event`，并返回取消退出码。

### 6.3 `RunnerContext`：运行期依赖

```python
@dataclass
class RunnerContext:
    request: RunnerRequest
    output_dir: Path
    control: RunControl
```

限制：

- 不包含 Qt、数据库连接、HTTP Session、任务队列客户端。
- Runner 如果需要浏览器/API 客户端，应在其内部按任务生命周期创建和释放。
- 后续 Worker 相关对象不得直接添加到 Context；Worker 应通过 Host 输入/输出边界与 Runner 解耦。

### 6.4 `RunnerEvent`：统一执行事件

```python
@dataclass(frozen=True)
class RunnerEvent:
    protocol_version: int
    task_id: str
    tool_id: str
    seq: int
    timestamp: str
    type: Literal[
        "log",
        "progress",
        "item",
        "artifact",
        "metric",
        "error",
        "complete",
    ]
    payload: Mapping[str, Any]
```

事件最小 payload 约定：

| 类型 | 作用 | 最小 payload |
|---|---|---|
| `log` | 业务日志 | `level`, `message` |
| `progress` | 任务进度 | `completed`, `total`, `message` |
| `item` | 单条结果 | `item`；P0 默认关闭 |
| `artifact` | 文件/目录产物 | `name`, `path`, `media_type`, `size_bytes` |
| `metric` | 资源/速率采样 | `name`, `value`, `unit` |
| `error` | 非终态或终态错误详情 | `code`, `message`, `retryable` |
| `complete` | 唯一终态事件 | `status`, `duration_ms`, `artifact_count` |

### 6.5 终态规则

为避免重复成功/失败状态，采用以下严格规则：

1. Runner 只产出业务事件：`log`、`progress`、`item`、`artifact`、`metric`。
2. Host 捕获校验失败、取消、异常，并负责产出 `error`。
3. Host **总是**在最后产出一个且仅一个 `complete` 事件。
4. `complete.payload.status` 枚举为：`succeeded`、`failed`、`cancelled`。
5. `complete` 是 stdout 中最后一条事件。

这样未来 Worker 只需以最后一个 `complete` 为任务终态，不必猜测异常是否已经结束。

### 6.6 `CollectorRunner` 协议

```python
from typing import Iterator, Protocol


class CollectorRunner(Protocol):
    """无 UI 的单任务执行器。"""

    def validate(self, request: RunnerRequest) -> None:
        """参数或环境不满足要求时抛出 RunnerValidationError。"""

    def run(self, context: RunnerContext) -> Iterator[RunnerEvent]:
        """产出非终态业务事件；不得直接输出 stdout。"""
```

异常类型：

```python
class RunnerError(Exception):
    code = "runner_error"
    retryable = False


class RunnerValidationError(RunnerError):
    code = "validation_error"


class RunnerCancelled(RunnerError):
    code = "cancelled"


class RunnerExecutionError(RunnerError):
    code = "execution_error"
```

### 7.4 Host 职责

Host 必须按以下顺序工作：

1. 读取并解析请求 JSON。
2. 验证 `protocol_version`、`task_id`、`tool_id` 与 JSON 可序列化约束。
3. 通过 manifest 找到 `runner_entrypoint`。
4. 导入并实例化 Runner。
5. 创建任务输出目录。
6. 调用 `runner.validate(request)`。
7. 迭代 `runner.run(context)`，规范化并输出每一个事件。
8. 对 `artifact` 事件检查路径存在、在允许目录内、大小可读取。
9. 捕获取消、参数错误、导入错误、业务异常。
10. 输出唯一 `complete` 事件，设置进程退出码。

### 7.5 退出码

| 退出码 | 语义 | Host 最终状态 |
|---:|---|---|
| 0 | 成功 | `succeeded` |
| 2 | 请求或参数校验失败 | `failed` |
| 3 | Manifest/Runner 导入失败 | `failed` |
| 4 | Runner 执行异常 | `failed` |
| 5 | 产物协议错误 | `failed` |
| 130 | 用户取消 / SIGINT | `cancelled` |

### 7.6 stdout / stderr 强约束

- `stdout`：只写 `RunnerEvent` JSONL，不打印 banner、进度条、调试文本、traceback。
- `stderr`：可写日志与异常栈；PyQt 中显示时应单独作为诊断区域，不混入机器事件流。
- 任一非 JSON 的 stdout 行都视为 Host 协议违规，测试必须失败。

## 8. Manifest 与工具发现改造

### 8.1 `ToolSpec` 扩展

建议将 `ToolSpec` 改成：

```python
@dataclass(frozen=True)
class ToolSpec:
    tool_id: str
    name: str
    category: str
    summary: str
    entrypoint: str
    implementation_path: str = ""
    tags: tuple[str, ...] = field(default_factory=tuple)

    # P0 新增，保持 optional 以兼容尚未迁移的工具
    runner_entrypoint: str | None = None
    runner_version: int | None = None
```

不在 P0 加入 `param_schema`、`capabilities`、`resource_group`、`session_modes` 等 P2/P3 字段，以免元数据扩张速度超过实际迁移速度。

### 8.2 Manifest 兼容规则

现有 manifest 保持有效。例如：

```json
{
  "tool_id": "xlsx_merge",
  "name": "XLSX 文件合并",
  "category": "数据处理",
  "summary": "选择文件夹，默认只合并文件名包含 keyword 的 .xlsx 文件，并重新生成连续序号。",
  "entrypoint": "src.processing.windows.XlsxMergeWindow",
  "runner_entrypoint": "src.processing.runners.XlsxMergeRunner",
  "runner_version": 1,
  "implementation_path": "processing/xlsx_merge.py",
  "tags": ["xlsx", "merge", "keyword", "data"]
}
```

字段规则：

| 字段 | P0 是否必填 | 说明 |
|---|---|---|
| `tool_id` | 是 | 全局唯一、稳定，不可因显示名称变化而修改 |
| `name` / `category` / `summary` | 是 | UI 展示元数据 |
| `entrypoint` | 是 | 保持现有 PyQt 入口兼容 |
| `runner_entrypoint` | 否 | 试点工具必须有；其余工具可缺省 |
| `runner_version` | 否 | 试点工具填 `1` |
| `implementation_path` | 否 | 继续用于健康检查 |
| `tags` | 否 | UI 搜索使用 |

### 8.3 `discovery.py` 改造

必须完成：

- 继续扫描现有的 `SCAN_DIRS`。
- 保留 JSON、必填字段、重复 `tool_id` 检查。
- 读取可选 `runner_entrypoint`、`runner_version`。
- 对可选 Runner 字段做格式检查：
  - `runner_entrypoint` 是非空字符串或 `null`。
  - `runner_version` 是正整数或 `null`。
- 提供 `discover_tool_map()`：返回 `dict[str, ToolSpec]`。
- 不在发现阶段导入 Runner；导入只在 `--check` 或 Host 执行时发生，避免启动主界面时引发依赖或副作用。

### 8.4 `registry.py` 改造

目标是过渡兼容，而不是继续手写清单。

推荐做法：

```python
# src/studio/registry.py
from src.studio.discovery import discover_tools


def get_tools() -> list[ToolSpec]:
    tools, errors = discover_tools()
    if errors:
        # 记录错误，但让调用方决定是否阻断
        ...
    return tools
```

禁止保留：

```python
TOOLS = [ToolSpec(...), ToolSpec(...), ...]
```

如现有调用点短期必须使用 `TOOLS`，可先提供只读兼容对象或在应用启动时注入，但应在 P0c 内将调用点改为 `get_tools()` / `discover_tool_map()`。

### 8.5 `tool_runner.py --check` 输出

检查结果应覆盖 UI 与 Runner 的就绪性：

```json
{
  "tool_id": "xlsx_merge",
  "implementation_path": "processing/xlsx_merge.py",
  "script_exists": true,
  "ui_entrypoint": "src.processing.windows.XlsxMergeWindow",
  "ui_importable": true,
  "runner_entrypoint": "src.processing.runners.XlsxMergeRunner",
  "runner_importable": true,
  "runner_ready": true
}
```

规则：

- `runner_entrypoint` 缺省时，`runner_ready = false`，但工具发现不失败。
- 对试点工具，`runner_ready` 必须为 `true`。
- `--check-all` 应以每行一个 JSON 输出，便于 CI 消费。

## 9. 试点工具改造方案

### 9.1 试点选择原则

第一批不追求功能覆盖，而追求验证执行边界：

| 试点 | 类型 | 验证问题 | 风险 |
|---|---|---|---|
| `xlsx_merge` | 纯本地数据处理 | JSON 输入、进度、产物、错误、CLI 基线 | 最低 |
| YouTube `profiles` 或 `context` | API 优先工具 | 外部 API、批量输入、限流/异常、XLSX 产物 | 中等 |

不建议将 `judge_aigc`、`anomaly_detection`、TikTok/X/Instagram/Facebook 作为 P0 首批试点：前者受模型配置或依赖完整性影响，后者会被浏览器登录态、页面变化和平台风控干扰。

### 9.2 `xlsx_merge` 改造

#### 9.2.1 拆分目标

将当前文件合并逻辑拆为：

```text
src/processing/xlsx_merge.py
├── discover_xlsx_files(...)
├── merge_xlsx_files(...)
└── MergeResult

src/processing/runners.py
└── XlsxMergeRunner

src/processing/windows.py
└── XlsxMergeWindow：仅收集 UI 参数、显示事件
```

#### 9.2.2 建议服务接口

```python
@dataclass(frozen=True)
class MergeResult:
    output_path: Path
    input_file_count: int
    written_row_count: int


def merge_xlsx_files(
    *,
    input_dir: Path,
    filename_keyword: str | None,
    output_path: Path,
    on_progress: Callable[[int, int, str], None] | None = None,
    should_stop: Callable[[], bool] | None = None,
) -> MergeResult:
    ...
```

约束：

- 服务函数不得导入 PyQt。
- 由调用方决定输出路径；不得在服务内部硬编码窗口或弹窗行为。
- `on_progress` 只用于兼容 UI/Runner，不得要求 Qt signal。
- 输出文件必须写在 `context.output_dir` 内。

#### 9.2.3 Runner 行为

`XlsxMergeRunner` 至少输出：

1. `log`：输入目录、匹配文件数量。
2. `progress`：每读取/合并一个文件。
3. `artifact`：最终 XLSX。
4. Host 生成 `complete`。

### 9.3 YouTube API 工具改造

#### 9.3.1 建议试点：`profiles`

`profiles` 的优先级高于关键词全量搜索、浏览器回退、评论抓取等复杂工具，原因是它更适合验证：

- API Key 传递和环境校验；
- 多条 URL 批量处理；
- 单条失败不中断整批任务；
- XLSX 产物；
- 外部错误如何转换为 `error` / `log` / 最终状态。

#### 9.3.2 改造原则

```text
YouTubeProfilesWindow
  ├── 读取文本框 / TXT 文件
  ├── 组织 params
  └── 调用共享 service 或 Runner 适配层

YouTubeProfilesRunner
  ├── validate：API Key、输入列表、输出路径
  ├── run：逐个调用 profile service
  └── 产出 log/progress/artifact

profiles.py
  └── 仅保留 URL 解析、API 调用、字段标准化、XLSX 写入服务
```

#### 9.3.3 P0 的错误处理边界

- 单个频道 URL 失效、不可访问、API 返回可识别失败：输出 `log` 或非终态 `error`，继续处理其他输入。
- API Key 缺失、请求文件无效、输出路径不可写：`validate` 直接失败。
- Google API 临时错误：可标记 `retryable=true`，但 P0 不实现自动重试。
- 触发浏览器回退的复杂逻辑不应被强行纳入首个 API Runner；若当前功能默认回退，P0 仅保持原有行为，并在日志中标明实际执行模式。

### 9.4 旧回调函数桥接器

部分现有函数可能已经依赖：

```python
run_xxx_spider(
    ...,
    log_callback,
    finish_callback,
    stop_event,
    pause_event,
    config={...},
)
```

允许在 P0 使用 `legacy.py` 进行过渡：

```python
class LegacyEventBridge:
    def log_callback(self, message: str, level: str = "INFO") -> None:
        ...

    def progress_callback(self, completed: int, total: int, message: str = "") -> None:
        ...

    def finish_callback(self, *args, **kwargs) -> None:
        ...
```

但它只允许作为迁移工具，不得成为所有新 Runner 的默认实现。新改造的业务函数优先返回清晰结果或接受通用回调。

## 10. PyQt 兼容迁移

### 10.1 原则

P0 不改变用户已有的桌面工作流：

- 工具卡片仍可打开。
- 原有输入方式、配置弹窗、暂停/停止和 XLSX 输出继续可用。
- 迁移后的窗口只能减少业务耦合，不能降低原有功能。

### 10.2 两种允许的窗口接入方式

#### 方式 A：窗口调用共享业务服务（首选）

```text
Window -> 参数转换 -> merge_xlsx_files / fetch_youtube_profiles -> UI 更新
Runner -> 参数转换 -> merge_xlsx_files / fetch_youtube_profiles -> RunnerEvent
```

适合 `xlsx_merge` 等容易拆分的工具。

#### 方式 B：窗口调用本进程 Runner 适配器（过渡方案）

```text
Window -> create RunnerRequest -> Runner.run(context) -> Qt signal 转发事件
```

适合已有回调/线程控制较复杂，暂时不宜大拆业务函数的工具。

要求：

- UI 只负责将 `RunnerEvent` 映射到日志框、进度条、完成提示。
- `complete` 事件不能被 UI 再自行伪造。
- 窗口不得解析 Runner 的 stderr 作为业务结果。

### 10.3 不建议的迁移方式

禁止：

```text
PyQt Window -> subprocess runner_host -> 解析 stdout -> 再写回同一 UI
```

原因：P0 中 PyQt 与 Runner 在同一进程共享服务层更容易调试，也能保留原有停止/暂停行为。子进程 Host 是为 CLI 与未来 Worker 准备，不应强制替代每个桌面窗口。

## 11. 测试与验收

### 11.1 工具发现与元数据测试

新增或修改测试必须保证：

- 每个 manifest 都有唯一的 `tool_id`。
- 每个 manifest 都可解析为 `ToolSpec`。

### 11.2 Runner 契约测试

`test_runner_contracts.py` 至少覆盖：

### 11.3 Host 集成测试

用 subprocess 执行：

```bash
python -m src.core.runner.host --tool-id xlsx_merge --request-file ...
```

验证：

- 成功时退出码 `0`。
- stdout 每行均能 `json.loads()`。
- 最后事件为 `complete/succeeded`。

### 11.4 `xlsx_merge` 回归测试

### 11.5 YouTube Runner 测试

P0 的 CI 不依赖真实 API Key。使用依赖注入或 mock：

- Key 缺失时 `validate` 失败。

## 13. 风险控制与非目标

### 13.1 主要风险

| 风险 | 触发信号 | P0 应对 |
|---|---|---|
| Runner 与 PyQt 各自复制业务逻辑 | 同一工具有两份处理流程 | 优先抽服务层，UI/Runner 仅做适配 |
| 静态 registry 残留导致漂移 | 新增 manifest 后 UI、检查、Runner 结果不一致 | 所有查询改走 discovery；registry 仅兼容包装 |
| stdout 被旧代码污染 | subprocess 输出包含 print/日志前缀 | Host 捕获/约束，测试逐行 JSON 解析 |
| 旧回调签名不统一 | 迁移时需要大量特殊分支 | 仅对试点提供小桥接器，长期改为服务层 |
| 输出路径混乱 | artifact 找不到或跑到旧平台目录 | Host 提供每任务目录，试点工具必须遵守 |
| YouTube 外部 API 影响 CI | 无 Key、限流、网络波动导致测试失败 | CI mock，真实 API 仅手工 smoke test |
| 范围膨胀 | 开始写任务表、Web 页面、Worker | Commit 边界严格限制在 P0 |

### 13.2 P0 不应提前优化的问题

- Runner 事件的全量 item 高吞吐写入。
- 多任务共享浏览器与 session pool。
- 远程取消、断点续跑、失败重试、幂等键。
- 复杂参数 schema 与自动生成表单。
- Runner 运行时版本隔离和插件热升级。

这些问题应在 P1/P2 由真实单机任务链路和压测数据驱动，而不是在 P0 预设复杂方案。

## 14. P0 完成定义

P0 被视为完成，必须同时满足：

- [ ] 主工具发现只依赖 manifest；无人工维护的静态 TOOLS 列表。
- [ ] `ToolSpec` 和 manifest 支持可选 `runner_entrypoint`。
- [ ] `tool_runner.py --check` 能检查 UI 与 Runner 就绪性，且不启动 UI。
- [ ] `src/core/runner/` 存在稳定 Request、Context、Event、Runner、Host 契约。
- [ ] Host 可读取 JSON request、执行 Runner、以 JSONL 输出事件，并有明确退出码。
- [ ] stdout 最后一行始终为唯一 `complete` 事件。
- [ ] `xlsx_merge` 可无 UI 执行，并输出 artifact 事件。
- [ ] 一个 YouTube API 优先工具可无 UI 执行，并保留批量输入与 XLSX 产物能力。
- [ ] 两个试点工具的 PyQt 使用方式没有功能回退。
- [ ] Runner 契约、Host、manifest、试点工具、既有配置流的测试均通过。
- [ ] README 或开发文档已说明如何执行 `runner_host` 与 `--check-all`。

完成 P0 后，项目将具备进入 P1 的前提：FastAPI 控制中枢与单 Worker 不再需要绕开 PyQt 或重写每个采集器，而可以直接复用已经稳定的 Runner Host 协议。
