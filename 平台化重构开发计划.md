## 1. 计划原则

本计划不以“尽快做成生产级平台”为目标，而以“尽快看见系统压力边界”为目标。每个阶段都必须回答一个可测问题，并保留可复盘的数据。

核心原则：

- **先 Runner，后 Web**：先让工具脱离 PyQt UI 独立执行，再建设 Web 和 Worker。
- **先压测闭环，后治理能力**：P1/P2 先拿到任务吞吐、事件写入、Worker 扩散和队列堆积曲线。
- **先可比较，再自动化**：每次压测必须绑定 `experiment_id`、`run_id`、`batch_id`、`param_template_id` 和 `event_policy`。
- **先采样，后全量**：日志、指标、item 事件默认可限频和采样，避免事件系统制造假瓶颈。
- **PyQt 过渡期继续可用**：任何阶段都不应让现有桌面工具不可用。
- **重型基础设施后置**：NATS、Kafka、对象存储、Kubernetes 只在压测结果证明需要时引入。
- **checkpoint 只为压测服务**：只在长任务重跑成本影响实验判断时引入，不把可靠恢复作为主线。

## 2. 阶段总览

| 阶段 | 名称 | 核心交付 | 主要回答的问题 | 可行性 | 复杂度 |
|------|------|----------|----------------|--------|--------|
| P0 | 执行核心解耦 | 元数据修复、Runner SDK、runner_host、试点 Runner | 现有工具能否脱离 UI 批量执行 | 高 | 中 |
| P1 | 单机压力闭环 | FastAPI 控制中枢、单 Worker、Web JSON 任务、experiment/run/batch | 单机任务链路吞吐基线是多少 | 中高 | 中高 |
| P2a | 多 Worker 与批量调度压测 | `claim-tasks`、能力匹配、批量投放、队列和分发指标 | 横向扩张是否带来真实吞吐提升 | 中 | 高 |
| P2b | 参数模板与 schema 表单 | `param_templates`、schema 抽象、实验参数组合 | 参数变化如何影响吞吐和失败率 | 中 | 中高 |
| P3 | 浏览器资源与会话压力 | Chrome/Context/Tab/CDP 曲线、`session_mode` 对比 | 浏览器型工具先崩在哪一层 | 中 | 高 |
| P4 | 极限吞吐与基础设施对比 | 事件采样矩阵、PostgreSQL/Redis/NATS/Kafka 对比、成本曲线 | 调度层和数据层的极限在哪里 | 中低 | 高 |

阶段依赖：

```text
P0 Runner SDK
  -> P1 单机压力闭环
  -> P2a 多 Worker 调度压测
  -> P2b 参数模板和标准实验组合
  -> P3 浏览器资源与会话压力
  -> P4 极限吞吐与基础设施对比
```

## 3. 通用开发约束

### 3.1 命令约束

所有 Python 命令必须使用项目虚拟环境：

```powershell
.\.venv\Scripts\python.exe -m ruff check .
.\.venv\Scripts\python.exe test/test_discovery.py
.\.venv\Scripts\python.exe test/test_config_profiles.py
.\.venv\Scripts\python.exe src/studio/tool_runner.py --check --tool-id <tool_id>
```

### 3.2 兼容性约束

- 不删除现有 PyQt 入口。
- 不破坏现有 manifest 发现能力。
- manifest 是平台化工具发现的主真源，静态 `registry.py` 只作为兼容兜底。
- 不强制所有工具一次性迁移到 Runner。
- P0/P1 不要求浏览器工具支持 checkpoint。
- P1 不建设自动 schema 表单，只提供 JSON 参数编辑。
- P2a 不做 DAG、不做权限审计、不做生产级恢复。
- P3 不做严格 session 治理，只做会话模式压力对比。
- P4 不以功能完备为目标，只以是否拿到边界曲线为验收标准。

### 3.3 推荐模块

| 模块 | 用途 |
|------|------|
| `src/runner/` | Runner SDK、事件协议、旧回调适配器、`runner_host.py` |
| `src/control_hub/` | FastAPI 控制中枢、任务 API、实验 API、事件写入 |
| `src/worker_agent/` | Worker 注册、心跳、`claim-tasks`、子进程执行 |
| `src/platforms/*/runners.py` | 各平台 Runner 入口 |
| `src/platforms/*/schemas.py` | 参数 schema 和压测默认值 |
| `web/` | Web 管理端，P1 可先做最小可用界面 |
| `test/test_runner_*.py` | Runner 相关测试 |
| `test/test_control_hub_*.py` | 控制中枢测试 |
| `test/test_worker_*.py` | Worker Agent 测试 |
| `test/test_pressure_*.py` | 压测模型和指标口径测试 |

### 3.4 压测数据口径

所有阶段都围绕以下核心字段组织：

| 字段 | 用途 |
|------|------|
| `experiment_id` | 一组压测问题，例如“TikTok Tab 上限” |
| `run_id` | 一次具体压测运行，例如“1 Worker / 4 并发” |
| `batch_id` | 一批同类任务投放 |
| `param_template_id` | 参数模板或参数组合 |
| `resource_group` | `G1/G2/G3` 工具资源等级 |
| `session_mode` | `local_profile/shared_state/fresh_session`，API 工具可为空 |
| `event_policy` | 日志限频、指标间隔、item 事件模式 |

## 4. P0 开发计划：执行核心解耦

### 4.1 阶段目标

P0 的目标是让现有工具具备无 UI 执行入口，并输出统一事件流。P0 不建设 Web 平台，不做多 Worker，不追求所有工具迁移。

P0 要回答的问题：

- 现有 `SimpleToolWindow.run_task(...)` 能否抽出稳定的无 UI Runner？
- 当前 `log_callback/finish_callback/stop_event/pause_event/config` 模式能否被统一事件协议包住？
- manifest、窗口 `tool_id`、本地配置三者是否能对齐？

### 4.2 范围内

- 修复工具元数据一致性。
- 定义 `RunnerContext`、`RunnerEvent`、`CollectorRunner`。
- 实现 `runner_host.py`，支持命令行执行 Runner 并输出 JSONL。
- 扩展 `ToolSpec` 和 manifest 字段。
- 实现旧回调函数到 RunnerEvent 的适配器。
- 改造 2-3 个低风险试点 Runner。
- PyQt 继续可用，并逐步改为调用 Runner。
- 增加 Runner 单元测试和 `tool_runner.py --check` 检查。

### 4.3 范围外

- 不做 Web 管理端。
- 不做 PostgreSQL 任务表。
- 不做 Worker Agent。
- 不要求浏览器型工具立即迁移。
- 不要求 checkpoint 恢复。
- 不要求所有 item 结构化入库。

### 4.4 推荐试点工具

| 工具 | 原因 | 注意点 |
|------|------|--------|
| `xlsx_merge` | 本地处理、无外部限流、最适合验证 Runner 基线 | 当前窗口类缺少 `tool_id` 时需补齐 |
| YouTube API 类工具 | API 路径清晰，便于观察输出和错误事件 | 需要 API key，不适合作为无配置 CI 全量测试 |
| 另一个纯输入输出清晰的处理工具 | 用于验证不同工具形态的适配器 | 避开缺依赖或强外部服务工具 |
| `judge_aigc` | 可选试点，用于验证外部 LLM 任务事件 | 只在 DeepSeek 环境变量和限流条件明确时使用 |

`processing_anomaly_detection` 暂不作为默认试点，因为当前依赖声明存在 pandas/numpy 缺口。

### 4.5 任务拆分

#### P0.0 工具元数据一致性修复

开发任务：

- 扫描所有 `*.manifest.json`。
- 对比 manifest `tool_id` 与窗口类 `tool_id`。
- 补齐缺失的窗口类 `tool_id`。
- 明确 discovery 的工具数量和测试期望。
- 让 `tool_runner.py --check --tool-id <tool_id>` 输出新字段可用性。

验收：

- `test/test_discovery.py` 通过。
- 所有 manifest 可被加载。
- 所有试点工具的 `tool_id` 能从 manifest、窗口、配置中一致追踪。

#### P0.1 定义 Runner SDK

建议结构：

```python
class RunnerContext:
    experiment_id: str | None
    run_id: str | None
    batch_id: str | None
    task_id: str
    tool_id: str
    params: dict
    config: dict
    session_mode: str | None
    event_policy: dict
    checkpoint: dict | None
    stop_event: object
    pause_event: object

class RunnerEvent:
    task_id: str
    seq: int
    timestamp: str
    type: str
    payload: dict

class CollectorRunner:
    def validate(self, params: dict, config: dict) -> None: ...
    def run(self, context: RunnerContext): ...
```

事件类型：

| 类型 | 用途 |
|------|------|
| `log` | 普通日志 |
| `progress` | 进度 |
| `item` | 单条结果，可采样 |
| `artifact` | XLSX、JSONL、截图、HTML 快照 |
| `metric` | 资源、速率、并发 |
| `checkpoint` | 可选阶段快照 |
| `error` | 错误 |
| `complete` | 完成 |

验收：

- `RunnerEvent.to_dict()` 可 JSON 序列化。
- `event_policy` 有默认值。
- 事件 seq 单调递增。
- RunnerContext 能从 JSON 输入构造。

#### P0.2 实现 `runner_host.py`

建议命令：

```powershell
.\.venv\Scripts\python.exe -m src.runner.runner_host `
  --tool-id xlsx_merge `
  --task-json .\tmp\task.json `
  --params-json .\tmp\params.json `
  --config-json .\tmp\config.json
```

支持参数：

| 参数 | 必填 | 用途 |
|------|------|------|
| `--tool-id` | 是 | 工具 ID |
| `--task-id` | 否 | 未提供时自动生成本地任务 ID |
| `--experiment-id` | 否 | 压测实验 ID |
| `--run-id` | 否 | 压测运行 ID |
| `--batch-id` | 否 | 批次 ID |
| `--params-json` | 是 | 参数 JSON 文件 |
| `--config-json` | 否 | 配置 JSON 文件 |
| `--session-mode` | 否 | 浏览器工具会话模式 |
| `--event-policy-json` | 否 | 事件采样策略 |
| `--checkpoint-json` | 否 | 可选 checkpoint 输入 |

验收：

- 标准输出为 JSONL 事件流。
- 失败时输出 `error` 事件并返回非零退出码。
- 完成时输出 `complete` 和 `artifact` 事件。

#### P0.3 扩展 ToolSpec 和 manifest

建议新增字段：

```json
{
  "ui_entrypoint": "src.platforms.youtube.windows.YouTubeKeywordWindow",
  "runner_entrypoint": "src.platforms.youtube.runners.YouTubeKeywordRunner",
  "capabilities": ["api", "xlsx_output"],
  "resource_group": "G1",
  "supports_checkpoint": false,
  "supports_remote_worker": true,
  "session_modes": [],
  "param_schema": "src.platforms.youtube.schemas.keyword_config",
  "pressure_defaults": {
    "recommended_worker_slots": 4,
    "metric_interval_ms": 1000,
    "item_event_mode": "sampled"
  }
}
```

兼容策略：

- 旧 `entrypoint` 保留给 PyQt。
- 新 `ui_entrypoint` 可从旧 `entrypoint` 迁移。
- `runner_entrypoint` 在未迁移工具上允许为空。
- `resource_group` 默认 `G1`，浏览器重型工具标 `G2/G3`。

#### P0.4 实现旧回调适配器

开发任务：

- `log_callback(message)` 转换为 `RunnerEvent(type="log")`。
- `finish_callback(path)` 转换为 `artifact` 和 `complete`。
- 捕获异常转换为 `error`。
- 按 `event_policy.log_rate_limit_per_sec` 限制日志事件。
- 保留 `stop_event` 和 `pause_event` 语义。

验收：

- 旧采集函数不需要一次性重写内部逻辑。
- 适配器可收集日志、产物、错误和完成状态。
- 过密日志可被限频，不影响 Runner 完成。

#### P0.5 改造试点 Runner

每个试点 Runner 至少包含：

- `validate(params, config)`。
- `run(context)`。
- 从 `context.params` 读取输入。
- 从 `context.config` 读取配置。
- 输出 `log/progress/artifact/complete`。
- 失败时输出 `error`。

验收：

- 试点工具可通过 `runner_host.py` 运行。
- PyQt 仍能运行同一工具。
- 产物路径被 RunnerEvent 捕获。

#### P0.6 PyQt 兼容集成

开发任务：

- PyQt 窗口保留现有字段和配置弹窗。
- 试点工具窗口调用 Runner，而不是直接调用旧采集函数。
- 运行状态、暂停、停止仍按原有 UI 体验工作。

验收：

- `main.py` 可启动。
- 试点工具在 PyQt 中行为不退化。
- 未迁移工具仍走旧 `run_task`。

#### P0.7 测试补充

建议测试：

```powershell
.\.venv\Scripts\python.exe test/test_discovery.py
.\.venv\Scripts\python.exe test/test_config_profiles.py
.\.venv\Scripts\python.exe -m ruff check .
```

新增测试：

- `test/test_runner_event.py`
- `test/test_runner_context.py`
- `test/test_runner_host.py`
- `test/test_manifest_runner_fields.py`

### 4.6 P0 验收标准

- manifest、窗口类、配置的 `tool_id` 口径一致。
- 至少 2 个试点工具可无 UI 执行。
- Runner 输出 JSONL 事件。
- PyQt 入口继续可用。
- 日志限频和 artifact 捕获可验证。
- P0 能给 P1 提供可批量调用的 Runner。

## 5. P1 开发计划：单机压力闭环

### 5.1 阶段目标

P1 的目标是完成 Web、控制中枢、单 Worker、Runner 的最小闭环，并跑出第一条单机吞吐基线。

P1 要回答的问题：

- 单 Worker 单机链路能承载多少任务并发？
- 事件写入 TPS 和日志推送是否提前成为瓶颈？
- 产物落盘是否明显拖慢任务完成时间？

### 5.2 范围内

- FastAPI 控制中枢。
- PostgreSQL 最小 schema。
- experiment/run/batch 元数据。
- 单 Worker Agent。
- `claim-tasks` 单机版。
- Web 最小任务中心。
- JSON 参数编辑器。
- 实时日志和基础指标。
- 产物下载。

### 5.3 范围外

- 不做自动 schema 表单。
- 不做多 Worker 调度。
- 不做权限、审计、通知、DAG。
- 不做严格恢复。
- 不把所有 item 全量入库作为默认目标。

### 5.4 任务拆分

#### P1.1 新增控制中枢项目结构

建议结构：

```text
src/control_hub/
  app.py
  settings.py
  db.py
  models.py
  schemas.py
  services/
    experiments.py
    tasks.py
    workers.py
    events.py
    artifacts.py
  api/
    tasks.py
    workers.py
    tools.py
    experiments.py
    artifacts.py
```

验收：

- FastAPI 应用可启动。
- `/health` 返回运行状态。
- 基础配置可从环境变量读取。

#### P1.2 建立 PostgreSQL schema

第一版表：

| 表 | 用途 |
|----|------|
| `tools` | 工具元数据快照 |
| `tool_versions` | 工具版本 |
| `experiments` | 压测实验定义 |
| `experiment_runs` | 单次压测运行 |
| `param_templates` | P1 可先存 JSON，P2b 再扩 schema |
| `workers` | Worker 当前状态 |
| `worker_heartbeats` | 心跳和资源快照 |
| `tasks` | 任务主表 |
| `task_attempts` | 执行尝试记录 |
| `task_events` | RunnerEvent |
| `task_artifacts` | 产物 |
| `task_metrics` | 资源和吞吐指标 |

P1 可暂不建：

- `collected_items` 全量结构化入库。
- 严格 checkpoint schema。
- 复杂权限表。

验收：

- migration 可创建所有 P1 表。
- 表字段包含 `experiment_id/run_id/batch_id/event_policy`。
- 可按 `run_id` 查询任务和事件。

#### P1.3 实现任务 API

接口：

```http
POST /api/experiments
POST /api/experiments/{experiment_id}/runs
GET  /api/experiments/{experiment_id}/runs

POST /api/tasks
POST /api/tasks/batch
GET  /api/tasks
GET  /api/tasks/{task_id}
POST /api/tasks/{task_id}/cancel
POST /api/tasks/{task_id}/retry
```

任务创建要求：

- 支持直接传 JSON `params`。
- 支持 `event_policy`，未传时使用工具默认值。
- 支持 `run_id` 和 `batch_id`。
- 默认 `status=queued`。

验收：

- 能创建单任务。
- 能创建一批任务并绑定同一 `batch_id`。
- 能查询某个 `run_id` 下所有任务。

#### P1.4 实现 Worker Agent 单机版

职责：

- 启动时注册 Worker。
- 上报能力标签。
- 上报心跳和资源快照。
- 调用 `claim-tasks` 获取任务。
- 启动 `runner_host.py` 子进程。
- 读取 JSONL 事件并上报。
- 上传或登记产物路径。

单机版 `free_slots` 可固定为 1 或配置值。

验收：

- Worker 可以注册。
- Worker 可以领取任务。
- Worker 可以执行试点 Runner。
- Runner 退出后任务状态变为 `succeeded` 或 `failed`。

#### P1.5 实现 Worker API

接口：

```http
POST /api/workers/register
POST /api/workers/{worker_id}/heartbeat
POST /api/workers/{worker_id}/claim-tasks
POST /api/tasks/{task_id}/events
POST /api/tasks/{task_id}/artifacts
POST /api/tasks/{task_id}/complete
POST /api/tasks/{task_id}/fail
```

`claim-tasks` 请求：

```json
{
  "free_slots": 1,
  "capabilities": ["xlsx_output", "api"],
  "resource_snapshot": {
    "cpu_percent": 32.1,
    "memory_mb": 1024
  }
}
```

P1 响应可以只返回 0 或 1 条任务，但接口必须允许返回多条，给 P2a 压测批量分发留口径。

验收：

- `claim-tasks` 能按能力匹配任务。
- 领取任务后写入 `task_attempts`。
- 能计算 `queue_lag_ms` 和 `dispatch_lag_ms`。

#### P1.6 Web 管理端最小实现

页面：

- 任务列表。
- 创建任务。
- 批量克隆任务。
- JSON 参数编辑器。
- 实时日志面板。
- Worker 状态列表。
- 产物下载入口。
- run/batch 基础汇总。

P1 不做：

- 自动 schema 表单。
- 拖拽 DAG。
- 权限分组。
- 复杂报表。

验收：

- 能从 Web 创建任务。
- 能看到任务状态变化。
- 能看到日志和 artifact。
- 能按 `run_id` 查看任务数量、成功数、失败数、平均耗时。

#### P1.7 事件采样与指标

开发任务：

- 支持 `event_policy.log_rate_limit_per_sec`。
- 支持 `event_policy.metric_interval_ms`。
- 支持 `event_policy.item_event_mode`：`full`、`sampled`、`artifact_only`。
- Worker 定时上报 CPU、内存、磁盘。
- 控制中枢记录 `event_write_tps`。

验收：

- 同一 Runner 在不同 event policy 下事件量可比较。
- 过密日志不会压垮 P1 WebSocket。
- 能显示基础 `event_write_tps`。

### 5.5 P1 验收标准

- Web 可创建任务。
- 单 Worker 可执行试点 Runner。
- 任务状态、日志、产物可见。
- 能按 `run_id/batch_id` 复盘一次压测。
- 能看到 `queue_lag_ms`、`dispatch_lag_ms`、`run_duration_ms`、`event_write_tps`。
- 至少完成一次单 Worker 饱和测试，记录并发数、耗时、事件量和资源曲线。

## 6. P2a 开发计划：多 Worker 与批量调度压测

### 6.1 阶段目标

P2a 的目标是从单 Worker 扩展到多 Worker，优先测横向扩张是否真的带来吞吐提升。

P2a 要回答的问题：

- 1、2、4、8 台 Worker 时吞吐是否线性增长？
- 控制中枢分发延迟是否成为瓶颈？
- Worker 领取、心跳和事件写入是否互相抢资源？

### 6.2 范围内

- 多 Worker 注册。
- 能力匹配和资源组匹配。
- `claim-tasks` 批量领取。
- 批量任务投放。
- 队列深度、分发延迟、执行耗时统计。
- 轻量重试。
- 可选 checkpoint 记录，但不做恢复主线。

### 6.3 范围外

- 不做公平调度。
- 不做跨 Worker 精确恢复。
- 不做 session 池。
- 不做 DAG。
- 不做生产级调度治理。

### 6.4 任务拆分

#### P2a.1 Worker 能力注册增强

Worker 上报：

```json
{
  "worker_id": "worker_01",
  "hostname": "data-node-01",
  "tools": ["xlsx_merge", "youtube_keyword_mining"],
  "capabilities": ["api", "xlsx_output"],
  "resource_groups": ["G1"],
  "max_slots": 4,
  "current_slots": 0,
  "chrome_instances": 0,
  "tabs": 0
}
```

验收：

- Web 能看到每台 Worker 的能力。
- 调度器能按 `tool_id` 和 `resource_group` 过滤 Worker。

#### P2a.2 批量领取与调度器

调度策略：

1. `status=queued`。
2. Worker 具备目标工具。
3. Worker 有 `free_slots`。
4. 优先领取同一 `run_id` 下的任务，便于压测批次收敛。
5. 按 Worker 空槽一次返回 0 到 N 条任务。

验收：

- `free_slots=4` 时最多返回 4 条任务。
- 多 Worker 同时 claim 不会领取同一任务。
- 能记录每条任务的 claim 时间和 assigned Worker。

#### P2a.3 批量投放压测

Web 和 API 支持：

- 克隆 N 条任务。
- 基于关键词列表或输入文件拆分任务。
- 给所有任务绑定同一 `batch_id`。
- 设置目标并发和 Worker 数。

验收：

- 可以一次投放 100/500/1000 条 G1 任务。
- 能看到队列深度随时间变化。
- 能导出 run 级压测摘要。

#### P2a.4 指标汇总

新增汇总：

| 指标 | 说明 |
|------|------|
| `throughput_tasks_per_min` | 每分钟完成任务数 |
| `throughput_items_per_min` | 每分钟产出 item 数 |
| `queue_lag_p50/p95` | 排队延迟 |
| `dispatch_lag_p50/p95` | 分发延迟 |
| `worker_utilization` | Worker 槽位利用率 |
| `event_write_tps` | 事件写入速率 |
| `failure_rate` | 失败率 |

验收：

- Web 或 API 可查询 run 级汇总。
- 至少能比较 1、2、4 Worker 的扩容收益。

#### P2a.5 轻量重试

重试策略：

- 只支持失败后重跑。
- 不承诺续跑。
- `max_attempts` 默认 1，可配置为 2 或 3。
- 每次执行写入 `task_attempts`。

验收：

- 失败任务可手动重跑。
- 自动重试不会污染原 attempt 记录。
- run 级报表能区分任务数和 attempt 数。

### 6.5 P2a 验收标准

- 多台 Worker 可同时接入。
- `claim-tasks` 支持批量领取。
- 能完成多 Worker 扩散测试。
- 能量化吞吐增益、队列堆积、分发延迟。
- 能指出 P2a 当前瓶颈是 Worker、控制中枢、数据库事件写入，还是任务本身。

## 8. P3 开发计划：浏览器资源与会话压力

### 8.1 阶段目标

P3 的目标是画出浏览器型工具的资源边界，而不是先追求完整恢复性。

P3 要回答的问题：

- 每台 Worker 能承载多少 Chrome 实例？
- 每个 Chrome/Context 能承载多少 Tab？
- `local_profile/shared_state/fresh_session` 对启动成本、失败率、风控触发有什么影响？
- 浏览器、目标站风控、数据库写入，哪一个先成为瓶颈？

### 8.2 范围内

- 浏览器资源采样。
- `session_mode` 压测矩阵。
- G2/G3 工具试点。
- Tab 并发和任务并发阶梯压测。
- 失败后重跑和 checkpoint 成本对比。
- 风控、验证码、403、429 计数。

### 8.3 范围外

- 不做严格 session 池治理。
- 不承诺跨 Worker session 迁移。
- 不要求所有浏览器工具 checkpoint。
- 不做生产级风控规避系统。

### 8.4 任务拆分

#### P3.1 浏览器资源采样

采样字段：

- `chrome_instances`
- `contexts`
- `tabs`
- `cdp_connections`
- `browser_startup_ms`
- `page_open_ms`
- `memory_mb`
- `cpu_percent`

验收：

- Worker 心跳包含浏览器资源。
- run 级报表能画出 Tab 数和失败率关系。

#### P3.2 session_mode 实现与对比

模式：

| 模式 | 开发任务 |
|------|----------|
| `local_profile` | 继续使用本地 `user_data/` |
| `shared_state` | 支持注入 cookies/localStorage 快照 |
| `fresh_session` | 每次冷启动，不复用登录态 |

验收：

- 同一工具可按三种模式创建任务。
- 能记录启动耗时和失败率。
- API 工具允许 `session_mode` 为空。

#### P3.3 G2/G3 工具试点

推荐顺序：

1. 中等浏览器详情页工具。
2. TikTok 或 X 关键词类重型滚动工具。
3. Facebook 或 Instagram 主页滚动工具。

选择标准：

- 输入数据可控。
- 输出产物明确。
- 失败信号容易记录。
- 能逐步提高 Tab 或任务并发。

验收：

- 至少 1 类 G2/G3 工具有完整资源曲线。
- 能识别明显拐点。

#### P3.4 浏览器压力矩阵

示例矩阵：

| run | Worker | Chrome | Context | Tab | session_mode |
|-----|--------|--------|---------|-----|--------------|
| run_001 | 1 | 1 | 1 | 1 | local_profile |
| run_002 | 1 | 1 | 1 | 2 | local_profile |
| run_003 | 1 | 1 | 1 | 4 | local_profile |
| run_004 | 1 | 1 | 2 | 8 | shared_state |
| run_005 | 2 | 2 | 2 | 8 | fresh_session |

验收：

- 每个 run 有独立 `param_template_id` 和 `event_policy`。
- 能导出吞吐、失败率、启动耗时和资源使用。

#### P3.5 失败后重跑与 checkpoint 成本对比

开发任务：

- 对长任务记录可选 checkpoint。
- 比较失败后完整重跑与 checkpoint 续跑的总耗时。
- 只记录实验成本，不承诺完整恢复能力。

验收：

- 至少一个长任务完成成本对比。
- 文档能说明是否值得继续投入 checkpoint 改造。

### 8.5 P3 验收标准

- 至少 1 类浏览器工具有清晰资源上限曲线。
- 能比较三种 `session_mode` 的启动成本和失败率。
- 能指出瓶颈先落在浏览器、目标站风控、数据库写入还是 Worker 资源。
- 能给 P4 提供下一步基础设施对比依据。

## 9. P4 开发计划：极限吞吐与基础设施对比

### 9.1 阶段目标

P4 的目标是在确认工具和浏览器边界后，继续冲击调度与数据层的极限吞吐。P4 不以“上完整生产平台”为目标。

P4 要回答的问题：

- PostgreSQL 任务表和事件表能承受多高写入压力？
- Redis/NATS/Kafka 是否真的带来收益？
- 对象存储是否能缓解本地磁盘瓶颈？
- 控制中枢单实例是否足够，什么时候需要多实例？

### 9.2 范围内

- 事件写入频率提升。
- `full/sampled/artifact_only` 事件策略对比。
- PostgreSQL 批量写入优化。
- Redis Pub/Sub 实时推送对比。
- NATS JetStream 或 Kafka 小范围验证。
- 本地产物与 MinIO/S3 对比。
- 控制中枢单实例与多实例对比。
- 成本曲线和吞吐曲线。
- AI 日志摘要可作为观察辅助。

### 9.3 范围外

- 不做 RBAC。
- 不做审计。
- 不做告警和消息推送主线。
- 不做 DAG 编排。
- 不做 Agent 自动操作。
- 不默认引入 Kubernetes。

### 9.4 任务拆分

#### P4.1 事件写入压力测试

压测维度：

- `log_rate_limit_per_sec`
- `metric_interval_ms`
- `item_event_mode`
- 批量写入大小。
- task_events 索引数量。

验收：

- 拿到 PostgreSQL 事件写入 TPS 曲线。
- 能说明事件系统是否提前成为假瓶颈。

#### P4.2 队列和消息系统对比

对比：

| 方案 | 验证内容 |
|------|----------|
| PostgreSQL task table | 基线 |
| Redis Pub/Sub | 实时推送压力 |
| NATS JetStream | 持久事件流和高压分发 |
| Kafka | 超大吞吐和消息回放 |

验收：

- 至少完成 PostgreSQL 与一种消息系统对比。
- 能给出是否继续引入的明确结论。

#### P4.3 产物存储压力测试

对比：

- 本地目录。
- 延迟落盘。
- MinIO/S3。
- JSONL 与 XLSX 同步写入差异。

验收：

- 能判断磁盘是否拖慢采集吞吐。
- 能判断对象存储是否值得引入。

#### P4.4 控制中枢扩展实验

开发任务：

- 单实例控制中枢压测。
- 多实例控制中枢实验。
- WebSocket 推送压力观察。
- API 请求延迟统计。

验收：

- 有单实例吞吐上限。
- 有多实例收益或无收益结论。

#### P4.5 成本和容量曲线

输出：

- Worker 数量与吞吐关系。
- Chrome/Tab 数与失败率关系。
- 事件策略与写入 TPS 关系。
- 基础设施方案与维护成本对比。

验收：

- 形成一份容量建议。
- 能明确下一轮投入优先级。

### 9.5 P4 验收标准

- P4 不以功能完备为目标。
- 至少拿到一条数据库写入极限曲线。
- 至少完成一种消息系统或存储方案对比。
- 能明确哪些基础设施值得引入，哪些暂时不值得。

## 结论

本计划与 v1.4 架构保持一致：先让现有工具具备无 UI 执行能力，再用单 Worker 建立压力基线，然后通过多 Worker、参数模板、浏览器资源矩阵和基础设施对比逐层逼近系统边界。

评价标准不是“平台是否已经生产完备”，而是每个阶段是否更清楚地回答了一个压力边界问题，并能用可比较的 run/batch 数据复盘。
