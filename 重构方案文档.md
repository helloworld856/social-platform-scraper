## 1. 修订摘要

v1.0 初稿更偏向长期生产平台蓝图。本版改为“压力边界探索架构”，目标不是先追求稳定性、安全性或完备治理，而是尽快回答系统上限问题：单机能跑到什么程度，多 Worker 扩展后瓶颈在哪里，浏览器与数据库分别先崩在哪一层。

- 将目标从“稳态生产平台”调整为“压测优先的实验平台”。
- 保留四层架构：Web 管理端、控制中枢、Worker、数据层。
- 第一阶段优先抽离无 UI 的任务执行核心，而不是立即重写全部采集器。
- 控制中枢早期推荐 Python + FastAPI，降低和现有 Python 采集代码的集成成本。
- gRPC、Kafka、Kubernetes、Temporal 仍然作为高压场景选项，但不作为实验期前置依赖。
- checkpoint、会话池、故障迁移不再作为早期主目标，只在能帮助压测长任务或昂贵任务时引入。
- 补充落地可行性审计结果：P0 前置元数据一致性修复，P1 仅做 JSON 参数闭环，P2 先做多 Worker 扩散，再做参数体系抽象，P3/P4 重点转向吞吐、并发和资源边界测绘。
- 增加压测运行维度、事件采样策略和 Worker 批量领取语义，避免压测结果被日志频率、任务分发竞争或参数不可比性带偏。

## 2. 项目现状判断

当前项目是一个 PyQt5 桌面应用，核心形态如下：

| 领域 | 当前实现 | 判断 |
|------|----------|------|
| 主入口 | `main.py` 启动 PyQt 主窗口 | 桌面应用形态明确 |
| 工具发现 | 扫描 `*.manifest.json` | 已具备插件化雏形，应保留并扩展 |
| 工具隔离 | 主窗口通过 `QProcess` 启动 `tool_runner.py` | 已有进程隔离，可迁移为 Worker 子进程模型 |
| 工具 UI | 每个工具继承 `SimpleToolWindow` | 表单与执行逻辑耦合，需要拆分 |
| 任务执行 | UI 收集参数后在线程中调用 `run_task` | 只有本地任务概念，没有平台级 task_id |
| 任务输出 | 采集器直接写 XLSX 到 `output/` | 需要改为事件流 / 数据项 / 产物三类输出 |
| 配置 | 本地 JSON，支持默认值、全局配置、命名参数模板 | 可迁移为中心化参数模板与 schema |
| 浏览器 | Chrome CDP + `user_data/` 持久登录态 | 单机可用，跨机器会产生状态漂移 |
| 测试 | 已有 discovery、config、pause 等测试 | 可作为重构保护网，但缺少任务模型测试 |
| 元数据一致性 | 少量窗口类未声明 `tool_id`，静态 registry 与 manifest 全量工具不完全一致 | P0 前置修复，避免 Runner 和配置映射漂移 |
| 依赖声明 | `anomaly_detection.py` 使用 pandas/numpy，但当前 requirements 未声明 | 不作为 P0 默认试点，除非先补依赖 |

现有采集函数普遍采用如下接口形态：

```python
run_xxx_spider(
    ...,
    log_callback,
    finish_callback,
    stop_event,
    pause_event,
    config={...},
)
```

这说明第一阶段最重要的工作不是做 Web 页面，而是定义一个无 UI 的执行协议，让 PyQt、Worker 和未来 Web 调度都调用同一套核心能力。

落地时需要特别注意两类不一致：

- 工具发现以 manifest 为准，当前测试期望为 23 个工具；静态 `registry.py` 只覆盖部分预装工具，不能作为平台化唯一真源。
- 部分窗口类缺少 `tool_id`，会影响参数模板绑定、Runner 绑定和 Web 工具表单生成，应在 Runner SDK 开始前修齐。

## 3. 架构目标与非目标

### 3.1 核心目标

- **统一任务管理**：用平台级任务替代散落在各窗口里的本地执行。
- **保留现有工具资产**：尽量复用已有采集器、manifest、配置和输出工具。
- **渐进迁移**：PyQt 版本在过渡期继续可用，避免大爆炸式重写。
- **可观测执行**：任务日志、进度、产物、错误、资源使用统一上报。
- **压测优先**：优先回答吞吐、并发、浏览器资源、数据库写入、任务堆积的边界问题。
- **可比较实验数据**：每次压测必须能按 experiment / run / batch / worker / 参数模板回放和对比。
- **可扩展 Worker**：多台数据机通过 Worker Agent 接入控制中枢，重点验证横向扩张是否真的带来吞吐提升。

### 3.2 非目标

以下能力不作为第一阶段目标：

- 细粒度权限和多租户。
- 完整审计和合规设计。
- 全量高可用、自动故障转移、严格幂等。
- 所有采集工具立即支持精确断点续采。
- 所有数据一开始就入库并完全替代 XLSX。
- 生产级安全边界、密钥体系、会话治理。
- 在事件写入还没成为明确瓶颈前，不追求所有 item 都实时结构化入库。

## 4. 总体架构

系统仍分为四层，但本版以探索压力边界为标准组织职责：

| 层级 | 名称 | 实验期职责 | 压测重点 |
|------|------|------------|----------|
| L1 | Web 管理端 | 创建任务、批量投放、观察日志、查看指标 | 队列深度、事件速率、任务堆积 |
| L2 | 控制中枢 | 任务 API、Worker 注册、任务分发、事件汇聚 | 调度吞吐、事件写入、状态更新频率 |
| L3 | Worker Agent | 拉取任务、执行 Runner、管理浏览器与并发 | CPU、内存、Chrome 数、Tab 数、CDP 连接数 |
| L4 | 数据层 | PostgreSQL + 本地产物，可按瓶颈切入缓存、队列和对象存储 | 事件写入 TPS、产物落盘速度、查询压力 |

```
┌──────────────────────────────────────────────────────────────┐
│                         Web 管理端                            │
│  任务中心 │ Worker 状态 │ 工具配置 │ 日志查看 │ 产物下载        │
└─────────────────────────────┬────────────────────────────────┘
                    REST API / WebSocket
┌─────────────────────────────┴────────────────────────────────┐
│                         控制中枢                              │
│  Task API │ Scheduler │ Worker Registry │ Event Ingest │ Config │
└─────────────────────────────┬────────────────────────────────┘
               HTTP/WebSocket 或 队列协议（按阶段演进）
┌─────────────────────────────┴────────────────────────────────┐
│                    Worker Agent / Runner SDK                  │
│  任务拉取 │ 自检 │ 子进程执行 │ 事件上报 │ 本地缓存 │ 会话注入    │
└─────────────────────────────┬────────────────────────────────┘
                         本地采集工具
┌─────────────────────────────┴────────────────────────────────┐
│                         数据层                                │
│ PostgreSQL │ 本地产物 │ Redis/NATS(可选) │ MinIO/S3(可选) │ 指标采样 │
└──────────────────────────────────────────────────────────────┘
```

## 5. 关键架构决策

### 5.1 优先做 Runner SDK，而不是优先做 Web

现有工具的执行逻辑仍被 PyQt 窗口包装。若直接做 Web，后端仍无法快速批量调用工具。因此第一优先级仍是抽象统一 Runner，但目标是尽快形成可压测的批量执行接口：

```python
class RunnerContext:
    experiment_id: str | None
    run_id: str | None
    batch_id: str | None
    task_id: str
    params: dict
    config: dict
    session_mode: str | None
    event_policy: dict
    checkpoint: dict | None
    stop_event: object
    pause_event: object

class RunnerEvent:
    type: str  # log | progress | item | artifact | checkpoint | error | complete | metric
    payload: dict

class CollectorRunner:
    def validate(self, params: dict, config: dict) -> None:
        ...

    def run(self, context: RunnerContext) -> Iterator[RunnerEvent]:
        ...
```

PyQt 窗口、Worker Agent、命令行检查工具都应调用同一 Runner。

### 5.2 控制中枢早期使用 Python / FastAPI

v1.0 建议 Go 或 Node.js。修订后建议：

- **第一阶段**：Python + FastAPI。
- **原因**：与当前 Python 采集器、Playwright、配置系统、测试体系一致，迁移成本最低。
- **实验期原则**：优先用 Python 跑出瓶颈曲线，再决定是否需要 Go 或专门网关。

### 5.3 第一版通信不用强制 gRPC

gRPC 双向流适合后续高吞吐 Worker 通信，但实验期应优先选择实现成本最低、最容易打压测的协议：

| 阶段 | 推荐通信 |
|------|----------|
| P1 | Worker 轮询 REST 拉任务，REST 上报事件 |
| P2 | WebSocket 保持 Worker 在线状态和批量控制 |
| P3/P4 | NATS JetStream / gRPC 按压力结果决定是否引入 |

理由：当前最大的未知数是浏览器型采集器的执行上限，不是协议优雅程度。

### 5.4 Kafka 和 Kubernetes 后置

Kafka、Kubernetes 适合高吞吐和大规模部署，但实验期不应提前背上它们的运维复杂度：

- 第一版消息层可使用 PostgreSQL 任务表；Redis Pub/Sub 只作为实时推送增强。
- 需要消息持久化和回放时，优先评估 NATS JetStream。
- 单机或小团队部署优先 Docker Compose。
- Kubernetes 只在“单机和多机物理 Worker 都压不出真正瓶颈”之后再引入。

### 5.5 checkpoint 只在对压测有帮助时引入

实验版不以完整恢复为目标。checkpoint 仅在以下场景优先：

- 单个任务耗时很长，重跑成本过高。
- 需要测量“中途中断后继续”对吞吐的影响。
- 需要保留高价值任务的局部进度。

多数实验任务允许失败后重跑，不强求故障迁移。

### 5.6 P2 拆分为调度阶段和配置阶段

v1.1 将“多 Worker 与参数模板体系”放在同一阶段，但落地审计后判断体量偏大。实验版仍拆分阶段，但拆分的核心目的是更快分别测出“调度边界”和“参数复杂度边界”：

| 子阶段 | 目标 | 说明 |
|--------|------|------|
| P2a | 多 Worker、能力注册、简单调度、批量投放 | 先看横向扩张带来的吞吐变化 |
| P2b | 参数模板、schema 表单、标准实验组合 | 再提高实验效率和批量操作效率 |

这样可以避免“调度还没压出上限、参数系统就先重构过度”。

### 5.7 P3 按工具资源边界推进

浏览器与 API 工具的边界差异极大，应按工具分级观察：

| 等级 | 含义 | 示例 |
|------|------|------|
| G1 | 轻量 API / 数据处理工具 | `xlsx_merge`、YouTube API 类工具 |
| G2 | 中等浏览器详情页工具 | 单链接详情、单主页扫描 |
| G3 | 重型滚动与多页并发工具 | TikTok 关键词、X 关键词、Facebook 主页滚动 |

P3 的目标是画出每类工具的资源和吞吐曲线，而不是先承诺恢复性。

## 6. Web 管理端

### 6.1 技术选型

| 项目 | 第一阶段推荐 | 备注 |
|------|--------------|------|
| 框架 | React 18 + TypeScript | 保留初稿选择 |
| 构建 | Vite | 足够轻量 |
| 数据请求 | TanStack Query | 任务列表、详情、配置天然适合 |
| 状态管理 | Zustand | 仅存 UI 局部状态 |
| 样式 | TailwindCSS 或 shadcn/ui | 偏后台管理，不做营销式页面 |
| 实时 | WebSocket | 任务日志、事件速率、Worker 状态推送 |

### 6.2 第一阶段页面

#### 6.2.1 任务中心

第一版必须具备：

- 创建任务。
- 批量克隆任务。
- 批量修改并发参数。
- 选择工具。
- 填写 JSON 参数。
- 查看任务状态。
- 查看实时日志。
- 下载任务产物。
- 停止任务。
- 重新运行失败任务。

P1 不建设自动 schema 表单。自动表单放到 P2b，在 Runner SDK、任务 API 和 Worker 执行链路压出明显瓶颈后再做。

任务状态建议：

```
created → queued → assigned → running → succeeded
                                 │
                                 ├→ cancel_requested → cancelled
                                 │
                                 └→ failed → retry_queued → running
```

#### 6.2.2 Worker 管理

第一版必须具备：

- Worker 列表。
- 在线 / 离线 / 忙碌 / 异常状态。
- 最近心跳时间。
- 当前运行任务。
- 已注册工具能力。
- Worker 当前并发数。
- 浏览器实例数 / Tab 数。
- Worker 日志摘要。

#### 6.2.3 工具配置

第一版必须具备：

- 查看工具列表。
- 查看基础参数模板。
- 编辑实验参数。
- 保存常用压测模板（P2b 后提供）。

P1 只保留“任务级 config JSON”能力。配置模板和 schema 表单放到 P2b。

#### 6.2.4 产物中心

第一版不强制建设完整数据仓库，但必须能管理产物：

- XLSX 文件。
- JSONL 日志。
- 截图或 HTML 快照。
- 任务运行元数据。
- 资源采样结果。

后续再把 XLSX 输出拆成结构化数据入库；实验期先保证压测结果可回放、可比对。

## 7. 控制中枢

### 7.1 第一阶段职责

控制中枢是平台的大脑，但第一版要克制：

- 提供任务 CRUD API。
- 维护任务状态机。
- 接收 Worker 注册和心跳。
- 向 Worker 分发任务。
- 接收 RunnerEvent。
- 将日志、进度、产物写入数据库。
- 通过 WebSocket 向 Web 管理端推送实时变化。
- 汇总吞吐、堆积和资源指标。

### 7.2 推荐模块

| 模块 | 职责 |
|------|------|
| API Server | REST + WebSocket |
| Task Service | 任务创建、状态变更、重试 |
| Worker Registry | Worker 注册、心跳、能力索引 |
| Scheduler | 简单分配策略 |
| Event Ingest | 接收日志、进度、产物、错误 |
| Template Service | 压测参数模板、参数组合、schema |
| Artifact Service | 管理 XLSX、JSONL、截图等产物 |
| Experiment Service | 管理 experiment / run / batch，汇总压测曲线 |

### 7.3 调度策略

实验版调度策略建议偏向“快速打满”：

1. **能力匹配**：Worker 必须声明拥有目标工具。
2. **空闲优先**：优先分配给当前运行任务数最少的 Worker。
3. **批量投放**：允许一次性向多个 Worker 推送同类任务，观察扩散速度。
4. **弱粘性**：不强求任务回原 Worker，除非任务显式依赖本地浏览器状态。
5. **批量领取**：Worker 上报 `free_slots` 后一次领取多条任务，用来观察 claim 延迟和分发吞吐。

成熟期再考虑严格公平性、网络亲和和会话亲和。

### 7.4 压测最小观测面

为了让压测结果可比较，P1 起至少保留以下观测面：

| 指标 | 用途 |
|------|------|
| `queue_lag_ms` | 任务从创建到被领取的等待时间 |
| `dispatch_lag_ms` | 控制中枢从可分发到 Worker 领取的延迟 |
| `run_duration_ms` | 单任务真实运行耗时 |
| `event_write_tps` | 控制中枢和数据库事件写入压力 |
| `item_rate` | 采集或处理结果产出速率 |
| `artifact_bytes` | 产物落盘和传输压力 |
| `worker_cpu/memory/disk` | Worker 资源拐点 |
| `chrome_instances/contexts/tabs/cdp_connections` | 浏览器资源边界 |
| `target_error_rate` | 目标站风控、验证码、403、429 等外部压力 |

事件上报需要有采样和批量写入策略：`log` 可以限频，`metric` 固定间隔上报，`item` 在 P1/P2 可先采样或批量落 JSONL。只有当结构化数据入库本身就是压测目标时，才打开全量 item 入库。

## 8. Worker Agent 与 Runner SDK

### 8.1 Worker Agent 职责

Worker Agent 是部署在数据机上的常驻进程：

- 启动时注册 Worker。
- 扫描本机可用工具。
- 上报能力标签和版本。
- 拉取或接收任务。
- 为每个任务启动独立子进程执行 Runner。
- 接收 RunnerEvent 并上报控制中枢。
- 维护本地任务缓存、日志和可选 checkpoint。
- 管理浏览器上下文、Tab 数和并发槽位。

### 8.2 从现有 QProcess 模型迁移

当前主窗口已通过 `QProcess` 启动 `tool_runner.py`。这个模型可以演进为：

```
PyQt Main Window
  └── QProcess: tool_runner.py --tool-id ...

Worker Agent
  └── subprocess: runner_host.py --task-id ... --tool-id ...
```

区别是：

- `tool_runner.py` 当前启动 QWidget。
- `runner_host.py` 未来启动无 UI Runner。
- Runner 输出 JSONL 事件流，而不是直接弹窗。

### 8.3 RunnerEvent 规范

建议所有 Runner 统一输出事件：

```json
{
  "task_id": "task_20260609_0001",
  "seq": 128,
  "timestamp": "2026-06-09T14:32:00+08:00",
  "type": "progress",
  "payload": {
    "completed": 683,
    "total": 1000,
    "message": "已处理 683/1000"
  }
}
```

事件类型：

| 类型 | 用途 |
|------|------|
| `log` | 普通日志 |
| `progress` | 进度更新 |
| `item` | 单条采集数据 |
| `artifact` | 输出文件或截图 |
| `checkpoint` | 阶段快照（可选） |
| `error` | 执行错误 |
| `complete` | 任务完成 |
| `metric` | 资源采样、速率、并发信息 |

### 8.4 采集器改造策略

不要一次性改完所有工具。建议按“最容易压出边界”的顺序推进：

1. **数据处理工具优先**：XLSX 合并、AIGC 判断最容易快速形成 Runner 基线。
2. **API 类工具其次**：YouTube API 类工具容易观察单机吞吐和批量 fan-out。
3. **浏览器采集工具后置**：TikTok、X、Instagram、Facebook 适合用来测 Chrome / Playwright 压力上限。

## 9. 数据层

### 9.1 PostgreSQL 主数据库

第一版主数据库只需要 PostgreSQL，目标是先测出其写入和查询极限。

核心表建议：

| 表名 | 用途 |
|------|------|
| `tools` | 工具元数据 |
| `tool_versions` | 工具版本 |
| `experiments` | 压测实验定义 |
| `experiment_runs` | 一次参数组合、Worker 数和资源策略的实验运行 |
| `param_templates` | 压测参数模板 |
| `workers` | Worker 注册信息 |
| `worker_heartbeats` | 最新心跳快照 |
| `tasks` | 任务主表 |
| `task_attempts` | 每次执行尝试、领取 Worker、开始/结束时间和退出结果 |
| `task_events` | RunnerEvent 事件表 |
| `task_artifacts` | 任务产物 |
| `task_metrics` | 资源采样、速率、并发曲线 |
| `checkpoints` | 可选，昂贵任务的阶段快照 |
| `collected_items` | 可结构化入库的数据项 |

### 9.2 去重策略

实验期不把去重作为主目标，只有在重复数据明显影响压测判断时才启用轻量去重。

可选方案：

1. **直接允许重复写入**：优先观察原始吞吐。
2. **数据库唯一约束**：当重复写入污染统计结果时再加唯一键。
3. **缓存快速过滤**：当数据库去重成为瓶颈时，再考虑 Redis Bloom Filter。

写入策略：

```sql
INSERT INTO collected_items (...)
VALUES (...)
ON CONFLICT (dedupe_key) DO NOTHING;
```

### 9.3 对象存储

MinIO / S3 适合：

- XLSX 产物。
- 截图。
- HTML 快照。
- 大型 JSONL 日志归档。
- 采集到的图片、视频等非结构化数据。

第一版可以先落本地目录，并在数据库记录路径。对象存储只在本地磁盘成为明显瓶颈时再引入。

### 9.4 TimescaleDB

TimescaleDB 只用于时序指标：

- Worker CPU、内存、磁盘。
- 心跳历史。
- 任务吞吐速率。
- 错误率趋势。
- 浏览器实例 / Tab / CDP 连接数随时间变化。

不建议把普通采集数据存进 TimescaleDB。

### 9.5 消息队列选择

| 场景 | 建议 |
|------|------|
| 第一版 | PostgreSQL task table + Worker 轮询 |
| 小规模实时推送 | Redis Pub/Sub |
| 需要持久事件流 | NATS JetStream |
| 超大吞吐和消息回放 | Kafka |
| 需要压测调度极限 | NATS 或自定义批量分发 |

### 10.4 压测阶段

| 阶段 | 能力 |
|------|------|
| P1 | 单 Worker 闭环，测单机任务链路吞吐 |
| P2a | 多 Worker fan-out，测分发速度和队列堆积 |
| P2b | 参数模板化，测不同参数组合的性能差异 |
| P3 | 浏览器资源边界，测 Chrome / Context / Tab / CDP 上限 |
| P4 | 极限吞吐与成本曲线，测数据库、消息和调度上限 |

### 10.5 推荐压测场景

建议优先覆盖以下场景：

1. **单 Worker 饱和测试**：固定 1 台 Worker，逐步提升同类任务并发数，观察 CPU、内存、Chrome 数、事件写入 TPS。
2. **多 Worker 扩散测试**：从 1 台扩到 2、4、8 台 Worker，观察吞吐是否线性增长。
3. **浏览器 Tab 上限测试**：对 TikTok / X 关键词类工具逐步提升标签页并发，测页面崩溃点和风控点。
4. **数据库写入压力测试**：提高事件上报频率，观察 PostgreSQL 写入 TPS、事务延迟、索引开销。
5. **产物落盘测试**：大量 XLSX / JSONL 同时输出，观察磁盘写入速度和文件句柄压力。
6. **长任务重跑成本测试**：对昂贵任务比较“失败后重跑”和“checkpoint 续跑”的总成本。
7. **事件采样策略测试**：对同一批任务切换 full / sampled / artifact_only，确认事件系统是否提前成为假瓶颈。

## 11. 浏览器资源与会话压力

### 11.1 当前问题

当前浏览器模型依赖本机 `user_data/`：

- 单机模式简单直接。
- 多 Worker 间天然不统一。
- Chrome profile、Context、Tab、CDP 连接数会共同构成资源瓶颈。

### 11.2 推荐方案

实验版不优先做严格 session 治理，而优先测三种模式：

| 模式 | 用途 | 关注点 |
|------|------|--------|
| `local_profile` | 单机或单 Worker 快速试跑 | Chrome profile 体积、冷启动时间 |
| `shared_state` | 多 Worker 分发时复用 cookies / localStorage | 注入耗时、状态漂移 |
| `fresh_session` | 完全冷启动压力测试 | 登录成本、风控触发率 |

实验期核心观测指标：

- 每个 Worker 可承载的 Chrome 实例数。
- 每个 Chrome 可承载的 Context 数。
- 每个 Context 可承载的 Tab 数。
- 不同模式下的任务启动耗时。
- 风控、验证码、403、429 的出现阈值。

### 11.3 过渡策略

第一阶段默认继续使用本地 `user_data/`，但在任务模型中保留 `session_mode` 字段，便于后续对比实验：

| 模式 | 说明 |
|------|------|
| `local_profile` | 继续使用本地 Chrome profile |
| `shared_state` | 使用外部注入的 cookies / localStorage |
| `fresh_session` | 完全冷启动，不复用任何登录态 |

API 或纯数据处理工具不需要浏览器会话时，`session_mode` 可以为空，不要把 `api_only` 混入浏览器会话模式枚举。这样可以在不重写全部工具的情况下，先画出会话模式对吞吐和风控的影响。

## 16. 分阶段落地路线

### P0：执行核心解耦

目标：让现有工具具备无 UI 执行入口，并能开始收集统一的压测事件。

- 先做工具元数据一致性修复：
  - 补齐窗口类 `tool_id`。
  - 明确 manifest 是工具发现的唯一真源。
  - 让 `tool_runner.py --check` 输出新增 Runner 字段。
  - 暂不选择依赖声明不完整的 `anomaly_detection` 作为默认试点。
- 定义 `RunnerContext`、`RunnerEvent`、`CollectorRunner`。
- 新增 `runner_host.py`，支持命令行执行工具 Runner。
- 扩展 manifest，加入 `runner_entrypoint`。
- 选择 2-3 个低风险工具试点：
  - `xlsx_merge`
  - YouTube API 类工具，如频道信息或上下文采集
  - 另选一个纯输入输出清晰、外部限流较少的处理工具；`judge_aigc` 仅在 DeepSeek 环境变量和外部限流条件明确时作为可选试点
- PyQt 窗口改为调用 Runner。
- 增加 Runner 单元测试。

验收标准：

- PyQt 原有使用方式不受影响。
- 试点工具可以无 UI 执行。
- Runner 能输出 JSONL 事件。
- 日志、产物、错误都能被统一捕获。
- 工具发现、参数模板绑定和 Runner 绑定均以 manifest + `tool_id` 为准。

### P1：单机平台闭环

目标：完成 Web + 控制中枢 + 单 Worker 的最小闭环，并能测单机任务吞吐。

- FastAPI 控制中枢。
- PostgreSQL 任务表。
- 建立 experiment / run / batch 最小元数据。
- Worker Agent 拉取任务并执行 Runner。
- Web 任务中心，创建任务时先使用 JSON 参数编辑器。
- 实时日志查看。
- 产物下载。
- 基础资源采样和事件速率统计。

验收标准：

- 能从 Web 创建任务。
- Worker 能执行任务。
- Web 能看到状态和日志。
- 任务完成后能下载 XLSX。
- 能看到基础资源和事件速率。
- 能按 run / batch 对比同一工具的吞吐、耗时和事件写入速率。

### P2a：多 Worker 与调度

目标：多数据机协作，优先测横向扩张是否真的带来吞吐提升。

- Worker 注册和能力匹配。
- 多 Worker 调度。
- Worker 按 `free_slots` 批量领取任务。
- 批量任务投放。
- 可选的轻量重试。
- 可选 checkpoint 记录。
- 结构化日志入库。
- 吞吐、队列深度、分发延迟统计。

验收标准：

- 多台 Worker 可同时接入。
- 任务能分配到具备能力的 Worker。
- 能量化横向扩张后的吞吐变化。
- 能看到分发延迟和队列堆积。

### P2b：参数模板与 schema 表单

目标：把参数系统从 PyQt 语境抽离出来，支持快速构造压测参数模板。

- 工具参数模板入库。
- 压测参数模板。
- 从 `FieldSpec` / `ConfigParam` 抽象 UI 无关 schema。
- Web 使用 schema 自动生成任务参数表单。

验收标准：

- 至少 P0/P1 试点工具可以使用 Web schema 表单创建任务。
- 能快速切换不同压测参数组合。
- PyQt 配置弹窗仍可正常使用。

### P3：浏览器资源与会话压力

目标：画出浏览器型工具的资源边界，而不是先追求完整恢复性。

- Chrome / Context / Tab / CDP 压力实验。
- 不同 `session_mode` 的对比实验。
- 重型滚动工具的并发上限测试。
- 失败后重跑与 checkpoint 的成本对比。
- 选出最适合高压分发的工具类型。

验收标准：

- 至少 1 类浏览器工具有清晰的资源上限曲线。
- 能量化 `local_profile`、`shared_state`、`fresh_session` 的启动成本差异。
- 能明确指出当前瓶颈先落在浏览器、目标站风控、还是数据库写入。

### P4：极限吞吐与扩展实验

目标：在确认工具和浏览器边界后，继续冲击调度与数据层的极限吞吐。

- 批量任务洪峰投放。
- 事件写入频率提升。
- PostgreSQL / Redis / NATS / Kafka 的对比压测。
- 控制中枢单实例与多实例对比。
- 成本曲线和吞吐曲线统计。

验收方式：

- P4 不以功能完备为目标。
- 以“是否拿到明确瓶颈曲线和扩容收益曲线”为验收标准。

## 17. 主要风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 采集器和 UI 耦合过深 | Web/Worker 无法复用 | 先做 Runner SDK |
| 直接上分布式导致范围爆炸 | 项目长期不可交付 | P0/P1 做最小闭环 |
| 浏览器型工具过早并发过高 | Chrome 崩溃或风控提前触发 | 分层提高 Context / Tab / 任务并发，记录拐点 |
| PostgreSQL 先成为热点 | 事件写入 TPS 限制整体扩张 | 降低事件频率、批量写入，必要时再引入 Redis/NATS |
| 事件上报过密制造假瓶颈 | 压测结果反映日志系统上限，而不是采集器或浏览器上限 | 为 `log`、`metric`、`item` 分别设置采样、限频和批量写入策略 |
| XLSX 直写拖慢吞吐 | 采集速度被磁盘写入拖住 | 将产物和事件解耦，必要时延迟落盘 |
| 参数空间过大 | 压测结果不可比较 | 先做参数模板和标准实验组合 |
| 工具元数据不一致 | Runner、配置、Web 表单绑定失败 | P0 先补齐 `tool_id`，manifest 作为唯一真源 |
| 依赖声明不完整 | Worker 环境自检误判或运行失败 | P0/P1 试点避开缺依赖工具，后续补 requirements 和自检 |

## 结论

本版架构的核心不是先做出“稳”的平台，而是先做出“能测出边界”的平台：

1. 保留 manifest 和工具资产。
2. 先修齐工具元数据，再抽离无 UI Runner。
3. 用 Python 控制中枢和单 Worker 跑出第一条吞吐基线。
4. 用多 Worker 扩散任务，判断横向扩张是否有效。
5. 对浏览器型工具做资源和会话压力实验，找出真正的瓶颈层。
6. 最后再决定是否值得引入更重的消息系统、对象存储或调度基础设施。

评价标准不是“是不是已经像生产平台”，而是“是不是已经能更快、更清楚地把系统打到边界并看见瓶颈”。
